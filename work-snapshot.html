Nucleus — work LLM wiki (snapshot)

Author: Rodrigo
Snapshot taken: 2026-05-23

This is a frozen snapshot. It documents the starting point that personal.html adapts from. The active version of this nucleus lives in a private system at work and will not be updated here — it is not part of this ecosystem. Only the entries the personal nucleus actually inherits patterns from are included; entries about work-specific ingest sources (Slack, Jira, Google Docs), the sensitivity register, and other work-only concerns have been omitted. The one anonymization is replacing a friend’s first name with “a friend and colleague” in Entry 0.

Preface

The work LLM wiki is an “AI Chief of Staff” for an engineering leader: a single web app with a navigation pane, a content pane, and an embedded terminal running Claude Code. It organizes daily intel, action items, and synthesis across initiatives. The system is built from off-the-shelf parts — Node+Express, vanilla JS, ttyd + xterm.js, Claude Code — and its content lives as HTML files in a directory tree, with the LLM as the steward who keeps the wiki maintained over time.

The entries below are AI-generated, written for AI consumption. They’re the captain’s-log decisions and lessons that the personal wiki inherits from. Internal cross-references like “Entry N” that point to omitted entries will be broken links — that’s expected for a snapshot.

Entry 0 — Foundation

Three threads landed close enough together to fuse into a thing worth building.

  1. The LLM-wiki concept itself. Karpathy’s gist had crossed my feed weeks earlier but didn’t click until a friend and colleague started evangelizing it. He showed me his working instance, which made the idea concrete in a way the gist alone hadn’t: an LLM as the steward of a directory of notes, cross-referencing, ingesting, refining over time. The core insight is that the maintenance burden — the reason humans abandon wikis — gets handed to the LLM, who doesn’t mind.
  2. HTML instead of Markdown for LLM-managed content. A post arguing that Claude Code generates substantially better artifacts when the target medium is HTML rather than Markdown. Karpathy’s framing assumes Markdown; this was the first evolution I could see making. HTML carries semantics Markdown can’t — classes, data-* attributes, structured listings, inline scripts — which downstream enables typed entities (action items, sensitivity, monitored docs) without bolting a database onto a notes app.
  3. ttyd + xterm.js as accessible infrastructure. a friend and colleague was running Claude Code in a terminal embedded inside his Obsidian window via an Obsidian plugin. I hadn’t realized browser-embeddable terminals were mature, widely-used technology — that ttyd serves a real shell over a WebSocket and xterm.js renders it competently in any iframe. This had been the gap for years: I’d wanted to mix terminal and web modalities in the same surface and didn’t know how to do it without bespoke engineering.

The synthesis. If I built a small bespoke web app instead of using Obsidian + plugins, I could put terminal and content in panes I controlled, store everything as HTML so structured-data affordances came for free, and have Claude Code edit both the content and the app itself from inside the app. That last part — the application building the application — was the move that made the project feel worth the effort, not just a marginal customization of someone else’s tool.

All three threads rested on accessible, off-the-shelf technology: HTML, ttyd and xterm.js, Claude Code’s filesystem tools. The work was assembly, not invention. That mattered because it meant the project was actually buildable in a few sittings, not a multi-month engineering investment.

The deeper question was what to organize the system around. Karpathy frames the LLM wiki as knowledge-shaped — query-driven retrieval over compounding notes. I wasn’t looking for a librarian. I was looking for an instrument for running my function as an engineering leader: daily intel, synthesis across initiatives, action tracking, prep for upcoming conversations. Same substrate (directory + LLM), different organizing principle. The “AI Chief of Staff” framing the preface mentions is what made everything that followed feel natural rather than forced.

Entry 1 — Minimal viewer, terminal, and CLI shape

The first phase. We needed a viewer for the content, a place for the agent to run, and a way to start and stop both. All three had to exist before any subsequent iteration would make sense.

What we built:

  • A Node + Express server on 127.0.0.1:3000 with two endpoints — /api/tree (walks the filesystem to produce a JSON nav tree, reading the first 1 KB of each HTML file to pull its <title> for the nav label) and /api/page (HTML pass-through, Markdown via markdown-it, CSV/TSV via papaparse, image bytes for image extensions, placeholder otherwise).
  • A separate ttyd server on 127.0.0.1:7681 serving a real shell over a WebSocket. xterm.js renders it inside an <iframe> in the frontend, so the viewer and the terminal live in the same browser tab from day one.
  • A vanilla-JS frontend in _app/public/: one index.html with two stacked panes (content top, terminal bottom, split by Split.js), one app.js, one styles.css. No build step.
  • A WebSocket at /ws so the CLI can broadcast “open this path” messages to any open browser tab.
  • A wiki CLI at _app/bin/wiki, symlinked into PATH: up, down, status, open, tree. up spawns both the node server and ttyd detached, writes PIDs and logs under _app/run/, probes both ports until they’re listening, and opens the browser; it fails fast if either port is already in use.

Why this shape: filesystem-as-substrate means git is the durable artifact without any extra plumbing — no database to lose, no schema to migrate, every tool already knows how to deal with files. Vanilla JS with no build means the app can be edited and reloaded without a rebuild loop, which turns out to be load-bearing later when the agent starts modifying the app from inside it. Local-only on 127.0.0.1 means no auth and no cloud surface; the whole system runs on a closed laptop. Terminal infrastructure was off-the-shelf and battle-tested, so we didn’t have to invent a shell-in-a-browser.

A few small decisions in here ended up mattering disproportionately. A leading underscore on a top-level name marks it as infrastructure (the app itself in _app/, runtime data, internal memory) and hides it from the content nav — so the nav shows what the operator authored, not everything in the repo. Every path arriving over the network gets resolved and verified to stay inside the wiki root before any file read; cheap on day one, painful to retrofit later. And the CLI is the only sanctioned way to manage lifecycle, so the operator never has to think about ports or PIDs — wiki up, wiki status, done.

Entry 4 — Workstream hierarchy and workstream-specific channels

The workstream pages introduced in Entry 2 were flat — one file per concern at the workstream root. That stopped scaling fast. Real workstreams have sub-workstreams; sub-workstreams have initiatives. Forcing them flat meant either too few pages (each one too dense to navigate) or too many (the root became a wall of one-line summaries).

The shape that worked: nested directories. A workstream is a directory with an index.html; a sub-workstream is a child directory with its own index.html; an initiative is a single file inside the sub-workstream, or a further-nested directory if it has its own internal structure. The viewer folds index.html into its parent directory in the nav — the directory takes the index page’s title and the index file doesn’t appear as a separate leaf.

Each workstream page can declare its own Slack channel subscriptions via a <ul class="channels"> block listing the channels relevant to that workstream specifically (a backend modernization page subscribes to a backend-modernization channel; the cancel-flow initiative subscribes to its project-specific channel). The daily flow aggregates these alongside the cross-cutting set from slack/ongoing.html and dedupes on channel name. When the same channel appears on multiple pages, deepest path wins — the most specific workstream gets the briefing summary.

The workstream-specific channels are where most of the actionable signal actually lives. Cross-cutting channels are important context; workstream-specific channels are where decisions get made and progress gets reported. Putting the subscription declaration on the workstream page itself means the watch-list isn’t a separate config that drifts out of sync — it’s right where the work is described.

Entry 8 — Viewer feel-finished pass

By the time the wiki had a few days of content, the viewer’s rough edges were obvious. A cluster of small moves tightened it up.

External links open in a new tab. The wiki tab is the operator’s focus surface; a docs.google.com or github.com link in a meeting summary shouldn’t replace it. The content-pane click handler routes any :- or //-prefixed href to window.open.

Tree nav shows page titles, not filenames. The tree-building code on the server reads the first 1 KB of each HTML file to extract <title>, falling back to the filename if absent. A directory with an index.html takes the index’s title as its own label, and the index file doesn’t appear separately in the tree. The nav reads like a topic hierarchy, not a file listing.

Canonical directory paths end with /. Without this, a relative href from workstreams/modernization/index.html like <a href="web/"> resolves against workstreams/ (the parent) instead of workstreams/modernization/ (the directory the index file conceptually is). Normalizing dir paths to trailing / in the frontend’s history.pushState calls fixes it. Small, but a constant source of broken links until handled.

Three pane-mode buttons. A top-bar group lets the operator snap to content-only, both panes, or terminal-only. The hidden pane stays in the DOM so the embedded terminal’s WebSocket stays alive — the agent session persists across mode switches.

Entry 11 — Sonnet for sub-agents

A model-choice calibration that lands once a few parallel-sub-agent pipelines exist: the per-meeting sub-agents from Entry 10, the channel scanners from Entry 3, ad hoc research deep-dives. They all do structured summarization — read a bunch of input, return a well-shaped output. Sonnet is roughly equivalent quality to Opus on this kind of task at roughly 5× lower cost.

The discipline: reach for Sonnet by default for sub-agents; pin Opus only when the task genuinely needs deeper reasoning (the main orchestrator, planning-heavy work, multi-step synthesis across heterogeneous inputs). Pin the model choice explicitly in the sub-agent’s prompt or Agent-tool call — don’t leave it to inheritance, because the inherited model is whatever the orchestrator is, which is usually Opus.

Entry 12 — Action items as first-class typed entities

The biggest single design move. Up to this point the system captured raw intel and synthesized it into workstream narratives, but actions surfaced from meetings and channels lived only as bullets inside their source documents. Aggregating “what do I need to do” required re-reading. The move: lift action items to a first-class typed entity that can live anywhere in the tree, with a stable markup convention that an aggregator endpoint can walk and return as JSON.

The markup. An <li> with class action-item inside a <ul class="action-items">. The item carries data-id (stable kebab-slug, ideally scoped by initiative), data-status (one of open, scheduled, blocked, done, dropped), data-owner, and optionally data-due (YYYY-MM-DD). The action text is the <li>’s body; an optional child <span class="note"> holds short context (blocker, dependency, link). The CSS keys off data-status — the same rules apply on initiative pages and on the aggregator view, so visual consistency is automatic.

Concretely:

<ul class="action-items">
  <li class="action-item"
      data-id="kebab-slug-scoped-by-initiative"
      data-status="open"
      data-owner="rod"
      data-due="2026-05-25">
    Imperative action text.
    <span class="note">optional context, blocker, or link</span>
  </li>
</ul>

The aggregator. A small parser (regex over the file tree, tolerant of attribute order) finds every action-item across the wiki and exposes them as JSON at /api/action-items. A dynamic viewer at action-items.html consumes the endpoint and renders the combined list with status-coded blocks. Orphan items — ones without a clear initiative home — live in inbox.html at the root. The parser strips <script> and <style> blocks before scanning, or string literals like 'class="action-item"' inside inline JS (the aggregator viewer itself, for instance) get mis-parsed as real items.

Why this is consequential. Items keep living next to the work they belong to (the right initiative page; inbox.html for orphans), but they can be queried as a single set. The operator has both views at once: in-context on the initiative page, plus a cross-cutting “everything that needs me” surface. This is what made the system shift from instrument for synthesis to instrument for action — the difference between knowing where things stand and knowing what to do next. It’s also the second compounding payoff of the HTML-over-Markdown choice (Entry 0): typed entities embedded in narrative text, walkable by a simple parser, no separate database.

Two disciplines that emerged with the markup:

  • Bias toward fewer items. Only track things that directly affect the operator — their own actions, or things they need others to do so they can act. It’s easy for ingest to surface every “next steps” bullet from every meeting as a candidate; if most candidates land as real items the list becomes noise. Treat ingestion-time proposals as candidates to reject by default.
  • Meeting “Next steps” stay as plain bullets. The Next-steps section of a meeting summary uses ordinary <ul><li>, not class="action-item" markup. Keeps raw transcript noise out of the aggregator. The proposal layer (Entry 13) is what lifts the few worth keeping into proper action items on the right initiative page.

Entry 13 — The proposal pattern

Introducing action items (Entry 12) raised a question: when /morning ingests yesterday’s meetings and notices new “next steps” or candidates for state changes, should it write them into the wiki directly? The answer is no. /morning never auto-edits initiative pages. It produces proposals only.

The pattern: each briefing’s “Action items — proposed” section has four subsections — new from yesterday’s meetings, state-change candidates (items that look done / unblocked based on intel since the last run), stale (open or scheduled with due dates in the past), and coming due soon. After the briefing opens, the orchestrating agent walks the operator through the proposals in the terminal pane conversationally — accept, edit, or reject each — and edits the appropriate initiative page or inbox.html on confirm.

Why proposals rather than auto-writes. The action-item list is a load-bearing source of “what needs me.” If the system auto-writes, the operator stops trusting it — one wrong item from a misread transcript and the whole surface becomes suspect. Proposals keep the operator in the loop on the decision and the system in the loop on the bookkeeping. Costs about thirty seconds per item; the trust is worth more than that.

The same proposal pattern extends to sensitivity (Entry 14): when ingest encounters a passage that might be sensitive but doesn’t match an existing register entry, it surfaces a candidates section rather than auto-marking or auto-extending the register. The discipline generalizes — for anything the system might get wrong in a way the operator would have to undo, propose rather than apply.

Entry 16 — xterm.js: pin the canvas renderer

A bug from the terminal pane that took longer to diagnose than to fix. Several times a day, letters in the embedded terminal would render with the wrong glyphs — an m drawn as an x, an h as a k. Source text was correct (visible by selecting and pasting); only the rendering was wrong. A hard-reload temporarily fixed it; corruption returned within an hour. Root cause: the default WebGL renderer in xterm.js maintains a glyph atlas in a GPU texture, and the atlas gets corrupted over long sessions in certain font / OS combinations.

The fix: pin xterm.js to its canvas renderer instead of WebGL, and pin an explicit monospace font stack. Both set via ttyd’s -t command-line options (-t rendererType=canvas and -t fontFamily="Menlo, Monaco, Consolas, monospace"). No code change to the wiki. Painless and durable; the corruption hasn’t returned since.

The lesson: when a third-party browser-embedded component has multiple rendering backends, the non-default one is sometimes the right pick for a long-lived session. WebGL is a sensible default for general usage; for “this terminal stays open all day,” canvas is the safer choice.

Entry 17 — tmux for shell persistence: tried and reverted

An experiment worth recording because it failed. The embedded terminal runs ttyd wrapping a shell; every time the WebSocket disconnects (page reload, network blip, server restart) the shell session ends. For most uses this is fine, but a Claude Code conversation in the terminal pane is more painful to lose. The natural move: wrap ttyd’s shell in tmux new-session -A so sessions survive disconnects.

It didn’t work out. Three concrete frictions: browser-native click-drag text selection broke (tmux mouse mode interfered); TUI redraw produced visible artifacts when split or resized; the configuration overhead (mouse mode tuning, set-clipboard wiring) added more operational surface than the session-persistence saved. Reverted to a plain zsh launch.

Why the cost-benefit didn’t justify it: Claude Code restart-and-resume is fast. Re-entering a session takes seconds; the agent reads the conversation history and is immediately back in context. The “session persistence” you thought you needed is mostly persistence of shell state (cwd, env vars, exported credentials), which is minor for this use case. Don’t reach for tmux unless the friction it introduces is genuinely smaller than the cost of restart.

Entry 18 — Same-origin terminal, pane hotkeys, focus that follows the visible pane

The pane-mode buttons from Entry 8 covered the visibility cases, but clicking a button to snap between content-only / split / terminal-only felt like enough friction that the operator drifted into not snapping at all. The fix should have been simple: a hotkey for “toggle sidebar” and another for “cycle pane mode.” It turned into a multi-move iteration because the obvious approach had hidden constraints.

First constraint: the embedded terminal is in an <iframe> pointed at ttyd (a different port). When focus is inside the iframe (which it almost always is, because the operator is typing to the agent), the parent page’s key listeners never fire. The iframe is cross-origin (different port = different origin), so the parent can’t attach listeners to its contentDocument either. The hotkeys silently fail in the exact case they’re most needed.

Fix: make the iframe same-origin by reverse-proxying ttyd through the wiki server. GET /term/* and its WebSocket upgrade get forwarded to ttyd on the loopback port; ttyd is launched with -b /term so the URLs in its served HTML carry the prefix the wiki expects. From the browser’s perspective the terminal is served by the wiki itself; the iframe is same-origin and the parent can manipulate its DOM.

Implementing the proxy surfaced a second issue. The wiki’s own WebSocket (/ws, the broadcast channel from Entry 1) used the ws library’s default {server, path} mode, which registers an upgrade listener that destroys sockets whose path doesn’t match. With the ttyd proxy in place, every /term/* WebSocket upgrade got destroyed by the /ws listener before the proxy handler could see it. Fix: switch the wiki’s WebSocket to noServer: true mode and write the upgrade listener by hand, with a path check that returns (rather than destroying the socket) when the path isn’t the wiki’s own. Lets other handlers own their paths.

With same-origin in place, the parent attaches a capture-phase keydown listener inside the iframe’s contentDocument. Capture phase fires before xterm.js sees the keys; the parent’s hotkeys intercept first, the rest pass through. Cmd/Ctrl + \ toggles the sidebar; Cmd/Ctrl + Shift + \ cycles the pane mode.

Third issue: focus. After cycling pane mode, focus should land in whatever pane just became active — and on transitions into split mode, in the pane that wasn’t previously visible (so “show me the terminal” doesn’t still require a mouse click to actually type). For content focus, focusing the content element works. For terminal focus, focusing the iframe’s contentWindow isn’t enough — xterm.js routes input through a hidden helper textarea (.xterm-helper-textarea), so focus has to land there specifically. Query for that element and call .focus() on it.

The operator now drives the layout from the keyboard regardless of which pane has focus, and the right pane gets focus after every transition. The “should have been simple” iteration ended up the right shape on the third or fourth try; each layer of the problem was discoverable only after the previous one was fixed. Three lessons emerge inside this iteration rather than as standalone entries: same-origin is required to manipulate an iframe’s document; noServer: true is required when coexisting with other upgrade handlers; xterm.js routes input through a hidden textarea, so focus the textarea not the iframe.

Entry 19 — The nucleus itself

The strange entry: this one describes the very document it lives in. After eighteen entries of iterations on the wiki, we stepped back and asked what kind of artifact this writeup actually was. The answer reshaped the document — and became the format the wiki is now exploring.

Earlier attempts were prose-shaped (“here’s how my wiki works”) or build-spec-shaped (“here’s enough that an LLM could rebuild it”). Both undersold what was interesting. The economics shift from Entry 0 — code now trivially reproducible from a clear description — means the canonical artifact of a software system can stop being the source code. What’s worth preserving as the first-class thing is the decisions and the reasoning behind them, in a form a coding agent can rebuild from and a human can lift from. That artifact needed a name.

We landed on nucleus: a single readable document holding the central state of a software system, distinct from the running code. The format is a captain’s log — one entry per coherent iteration, in roughly chronological order, each capturing what changed and why. Replay-shaped, not reference-shaped: you walk through how the system came together, you don’t look things up. Conventions and lessons live inline next to the iteration that produced them rather than in separate sections. Cross-references thread compounding payoffs and adjustments across entries.

Cross-references work through ordinary in-page anchor links (<a href="#entry-id">). That introduced a small viewer wrinkle worth noting: the wiki encodes the current page path as the URL fragment (#nucleus.html, etc.), so letting the browser natively handle in-page anchors would clobber the path encoding and 404 on the next reload. Fix: the viewer’s click handler intercepts #-prefixed hrefs and calls scrollIntoView programmatically, leaving the URL fragment alone. Small, easy to miss until cross-references actually start mattering.

Maintenance is the other half. A nucleus has to stay current as the system evolves, or it rots into stale description. That’s what /evening is for: scan what changed in the wiki since the last run, identify what crosses the threshold of nucleus-worthy, propose new entries (or amendments to existing ones), and apply on confirm. Same proposal discipline as Entry 13 — surface candidates conversationally; never auto-write. Where /morning keeps the wiki current with the world, /evening keeps the nucleus current with the wiki. The expected common case is “nothing nucleus-worthy today” — the nucleus shouldn’t grow every day.

The bet behind inventing the format at all is portability. Any software system whose code is now reproducible via agentic engineering could have a nucleus — one document holding the decisions and the journey, with a reflective routine to keep it current. The wiki is the first instance we have to point at; the format generalizes, or it doesn’t. Future runs of /evening will tell us.

Entry 22 — Patching iframe internals as a general technique

Entry 18 established that once the terminal iframe is same-origin (via the reverse proxy at /term/*), the parent can attach listeners inside the iframe’s contentDocument. The first use was capture-phase keydown for pane hotkeys. It turns out the same property unlocks a more general technique: patch third-party-library globals inside the iframe to bend their behavior without forking the library.

Concrete case: xterm.js’s web-links addon wraps every Cmd-clicked URL in a confirm() dialog (“Do you want to navigate to X? WARNING: This link could potentially be dangerous”) and then calls window.open() — both produce friction (the dialog every click, the popup blocker tripping because the call originates inside the iframe and can lose gesture-context inheritance). The fix lives entirely in the parent’s iframe-load handler: 

const win = termIframe.contentWindow;
const origConfirm = win.confirm.bind(win);
win.confirm = (msg) => {
  if (typeof msg === 'string' && msg.includes('Do you want to navigate to')) {
    return true;  // silently approve link-navigation prompts only
  }
  return origConfirm(msg);
};
win.open = (url, target, features) =>
  window.open(url, target || '_blank', features);  // delegate to parent

One subtle constraint that’s easy to miss: xterm.js’s sequence is e = window.open(); e.opener = null; e.location.href = uri — open a blank tab, detach opener, then set the URL. The wrapper must return a real Window reference and must not pass noopener in the features string (noopener makes window.open return null and the new tab stays on about:blank). The wrong move was caught on first try; fixed by dropping the noopener.

The general pattern: any time an embedded same-origin library calls a global the parent can reach, the parent can patch it. Useful for browser-side libraries whose authors weren’t thinking about your specific embedding context. Sister technique to capture-phase event interception from Entry 18 — both compound the value of having paid the same-origin cost once.

Entry 23 — Reverse-cycle hotkey for pane modes

Small affordance, worth stealing more than implementing from scratch. The pane-mode cycle from Entry 18 goes content-only → terminal-only → split, with focus on every transition landing in whichever pane just became visible. That rule is the right default — but when entering split, “the pane that just became visible” depends on where you came from, so the forward cycle always lands in split with focus on content (terminal was already visible; content is the pane that just appeared).

Sometimes you want to land in split with focus on the terminal instead — you’re reading something in the content pane and want to expand to drive the agent without losing your place. A complementary reverse-cycle hotkey gets this for free: bind a second combo to step the cycle backward, and the focus-follows-pane rule does the rest automatically. Approaching split from content-only, the terminal is the newly-visible pane and gets focus.

The lesson beyond the hotkey itself: focus-follows-pane is already directional. Reversing the cycle reverses which pane gets focus on transitions into split, without any new focus logic. The implementation is a few lines — a second function that steps the cycle index backward, plus a modifier check in the keydown handler. Modifier choice matters less than the conceptual pairing (forward / reverse share a key, vary by modifier so the gesture stays in your hand without re-anchoring).
