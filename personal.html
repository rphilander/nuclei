Preface

This document captures the decisions and reasoning behind my personal LLM wiki — adapted from the work version I built at Scribd (documented in nucleus.html, kept here as reference). The system is the same architecture (Node+Express, vanilla JS, ttyd, filesystem-as-database) but organized around personal workstreams rather than an engineering leader’s function. The primary audience is an LLM rebuilding or extending this system; the format is a captain’s log of iterations.

Entry 0 — Adaptation from work wiki

The work wiki was an “AI Chief of Staff” for an engineering leader at Scribd. The personal version keeps the architecture but changes the organizing principle and sources.

What carried over unchanged:

  • Node+Express server on 127.0.0.1:3000 with /api/tree, /api/page, /api/action-items endpoints.
  • Vanilla JS frontend with three-pane layout (sidebar, content, terminal). No build step.
  • ttyd reverse-proxied at /term/* for same-origin iframe access.
  • wiki CLI for lifecycle management.
  • Action items as first-class typed entities with data-status, data-owner, data-due attributes.
  • Declarative source subscriptions on workstream pages via <ul class="channels">.
  • Proposal pattern — system proposes, human confirms, never auto-writes.
  • The nucleus format itself.

What was dropped:

  • Sensitivity register and screen-share-safe toggle. No one else sees this wiki; the feature adds complexity for zero value.
  • Slack channel ingestion, Google Docs monitoring, Jira integration. Work-specific sources.
  • Brand directory and usage harvesters. Work-specific reference material.

What was adapted:

  • Workstreams reorganized around personal projects: TrueSpeech (language runtime), LinkedIn visibility, wardrobe reboot, AI developments.
  • /morning sources: Gmail MCP, Hacker News API, RSS feeds (Simon Willison), GitHub org activity (truespeech). Replaces Slack+Docs+Jira.
  • Hotkeys: Cmd/Ctrl+\ for sidebar, Cmd/Ctrl+Shift+\ for forward pane cycle, Cmd/Ctrl+Alt+\ for reverse pane cycle.

Entry 1 — Running in a Sprite VM

The wiki runs inside a Sprite VM rather than directly on a laptop. The Sprite machinery automatically forwards ports from the VM to the local MacBook, so localhost:3000 on the Mac reaches 127.0.0.1:3000 in the Sprite. This is transparent for the most part but introduces one constraint: the embedded terminal (via ttyd) must work entirely through the proxied port — there is no separate port 7681 forwarding.

Sprite checkpoints (not snapshots) are the rollback mechanism. Take checkpoints at milestones and before risky changes: sprite-env checkpoints create --comment "description".

Entry 2 — ttyd proxy: path preservation

First bug. The terminal iframe showed a 404 instead of a shell. Root cause: app.use('/term', proxy) in Express strips the /term prefix from req.url before handing to the proxy middleware. The proxy then forwarded GET / to ttyd, but ttyd was launched with -b /term and expects the prefix.

Fix: use pathFilter: '/term' in the proxy config instead of Express’s mount-path stripping. The full path reaches ttyd intact. A one-line conceptual change that was invisible until the 404 appeared.

Entry 3 — ttyd writable mode

Second bug. The terminal rendered correctly but ignored all keyboard input. The ttyd log said: The --writable option is not set, will start in readonly mode. ttyd v1.7.x defaults to read-only; the -W flag is required for interactive use. The work wiki had this; it was missed during adaptation.

Entry 4 — Terminal working directory

Third friction point. The shell inside the terminal started in _app/ instead of the wiki root (/home/sprite). This matters because claude --resume needs to run from the project root to find the correct session. Root cause: the wiki up command was cd’ing to $APP_DIR for the node server launch, and ttyd inherited that directory. Fix: cd to $WIKI_ROOT before launching ttyd, and launch the node server via absolute path instead.

Entry 5 — Tree walker: directories with only index.html

The nav tree was missing all four workstream directories. Root cause: the tree walker filtered out directories with zero children, but a workstream directory whose only file is index.html consumes that file as the directory’s title and skips it from the children list, leaving an empty array. Fix: track whether a directory has an index.html via a hasIndex flag, and include directories that have one even if they have no other children.

Entry 6 — Sprite services instead of raw PID management

The first version of the wiki CLI managed two background processes (node server, ttyd) by hand — spawning, writing PIDs to _app/run/, probing ports until listening. That works locally but the wiki was unreachable from outside the Sprite VM: the Sprite proxy at https://<sprite>.sprites.app routes external traffic to a service registered with --http-port, not to whatever happens to be listening on a port. No service, no route.

Fix: register ttyd and the node server as separate Sprite services with sprite-env services create. The wiki service declares --http-port 3000 (so the proxy routes there) and --needs ttyd (so dependency ordering is enforced). Two payoffs: external reachability, and auto-restart on crash — the supervisor brings the process back without the CLI having to track PIDs.

One gotcha: --args splits on commas. The ttyd font family arg fontFamily=Menlo,Monaco,Consolas,monospace got shredded into separate args; the start command became “Monaco\ Consolas\ monospace zsh” and ttyd tried to execvp a file named “Monaco\”. Dropped the font family from the service args (it’s cosmetic) and added fontSize=16 separately.

The wiki CLI (up/down/status) now wraps sprite-env services start/stop/list instead of doing the bookkeeping itself. Less surface area, fewer corners that go wrong.

Entry 7 — Browser-to-VM image bridge

ttyd is a text terminal — it cannot receive image pastes. But the wiki’s primary mode is “Claude Code running in the terminal pane,” and Claude Code reads images via file paths. So the operator’s real need is to get a screenshot from the MacBook clipboard into the VM filesystem, with the path on the clipboard for pasting into the terminal.

Solution: a drop/paste zone in the topbar. Cmd+V or drag-drop sends the image to POST /api/upload, the server writes it to _uploads/screenshot-<ts>.png, returns the absolute path, and the frontend copies that path to the OS clipboard via navigator.clipboard.writeText. The operator pastes the path into Claude Code and asks it to look.

Placement matters: the drop zone lives in the topbar header, not the content pane. Wiki pages are intended to be little applications — they may have their own paste handlers, drop targets, or interactive surfaces. Putting the bridge in topbar chrome keeps the content pane free for pages to be themselves.

Entry 8 — Server self-restart via process.exit

The terminal proxies through the node server. Restarting node from the terminal’s own Claude Code session kills the proxy connection, stranding the operator with no terminal and no easy way to recover. A topbar ⟳ button is needed so a restart can be triggered from the wiki page (which survives the restart) instead of the terminal (which doesn’t).

First attempt: POST /api/restart handler spawns sprite-env services restart wiki via execFile. Failed with open /sys/fs/cgroup/containers/sprite/svc.wiki/cgroup.procs: No such file or directory. A service cannot restart itself through sprite-env — the supervisor’s cgroup machinery doesn’t expect the target service to be the caller. The service hung in a half-broken state (PID reported as running, port not listening) until manually restarted.

Correct pattern: the handler returns 200, then calls process.exit(0). Sprite’s supervisor detects the exit and auto-restarts the service within ~2 seconds (see Entry 6). The frontend polls /api/tree until it gets 200, then reloads the terminal iframe. The operator stays in one browser tab; they need claude --resume in the new terminal session to pick up where they left off.

General lesson: when a process needs to restart itself, exit cleanly and let the supervisor handle it. Don’t reach back through the supervisor’s control plane from inside the supervised process.

Entry 9 — /checkpoint skill: commit messages as nucleus input

Two persistence layers run in parallel: git (durable, version- controlled) and Sprite checkpoints (full VM-state rollback). A milestone usually wants both, so doing them one-at-a-time is friction. The /checkpoint skill bundles them: survey changes, draft a commit message, commit, then create a Sprite checkpoint with the same subject line.

The interesting discipline isn’t the bundling — it’s the commit-message bar. Entry 0 establishes /evening reads git history to propose nucleus updates. That makes the commit message the nucleus’s primary input: every commit subject + body should capture the decision and reasoning, not just what files were touched. A “Fix bug” commit costs the nucleus a future entry it could have had.

The skill encodes the bar with positive and negative examples inline (good: “Register wiki as Sprite services instead of raw PID management” with the proxy-routing explanation; bad: “Update files”). When a change bundles multiple distinct decisions, the skill suggests splitting into separate commits so each decision gets its own message.
