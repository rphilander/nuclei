Preface

This document captures the decisions and reasoning behind my personal LLM wiki — adapted from the work version I built at Scribd (a frozen snapshot of the relevant work entries is in work-snapshot.html). The system is the same architecture (Node+Express, vanilla JS, ttyd, filesystem-as-database) but organized around personal workstreams rather than an engineering leader’s function. The primary audience is an LLM rebuilding or extending this system; the format is a captain’s log of iterations.

Entry 0 — Adaptation from work wiki

The work wiki was an “AI Chief of Staff” for an engineering leader at Scribd. The personal version keeps the architecture but changes the organizing principle and sources.

What carried over unchanged:

  • Node+Express server on 127.0.0.1:3000 with /api/tree, /api/page, /api/action-items endpoints. (work Entry 1)
  • Vanilla JS frontend with three-pane layout (sidebar, content, terminal). No build step. (work Entry 1, Entry 8)
  • ttyd reverse-proxied at /term/* for same-origin iframe access. (work Entry 18)
  • wiki CLI for lifecycle management. (work Entry 1)
  • Action items as first-class typed entities with data-status, data-owner, data-due attributes. (work Entry 12)
  • Declarative source subscriptions on workstream pages via <ul class="channels">. (work Entry 4)
  • Proposal pattern — system proposes, human confirms, never auto-writes. (work Entry 13)
  • The nucleus format itself. (work Entry 19)

What was dropped:

  • Sensitivity register and screen-share-safe toggle. No one else sees this wiki; the feature adds complexity for zero value.
  • Slack channel ingestion, Google Docs monitoring, Jira integration. Work-specific sources.
  • Brand directory and usage harvesters. Work-specific reference material.

What was adapted:

  • Workstreams reorganized around personal projects: TrueSpeech (language runtime), LinkedIn visibility, wardrobe reboot, AI developments.
  • /morning sources: Gmail MCP, Hacker News API, RSS feeds (Simon Willison), GitHub org activity (truespeech). Replaces Slack+Docs+Jira.
  • Hotkeys: Cmd/Ctrl+\ for sidebar, Cmd/Ctrl+Shift+\ for forward pane cycle, Cmd/Ctrl+Alt+\ for reverse pane cycle.

Entry 1 — Running in a Sprite VM

The wiki runs inside a Sprite VM rather than directly on a laptop. The Sprite machinery automatically forwards ports from the VM to the local MacBook, so localhost:3000 on the Mac reaches 127.0.0.1:3000 in the Sprite. This is transparent for the most part but introduces one constraint: the embedded terminal (via ttyd) must work entirely through the proxied port — there is no separate port 7681 forwarding.

Sprite checkpoints (not snapshots) are the rollback mechanism. Take checkpoints at milestones and before risky changes: sprite-env checkpoints create --comment "description".

Entry 2 — ttyd proxy: path preservation

First bug. The terminal iframe showed a 404 instead of a shell. Root cause: app.use('/term', proxy) in Express strips the /term prefix from req.url before handing to the proxy middleware. The proxy then forwarded GET / to ttyd, but ttyd was launched with -b /term and expects the prefix.

Fix: use pathFilter: '/term' in the proxy config instead of Express’s mount-path stripping. The full path reaches ttyd intact. A one-line conceptual change that was invisible until the 404 appeared.

Entry 3 — ttyd writable mode

Second bug. The terminal rendered correctly but ignored all keyboard input. The ttyd log said: The --writable option is not set, will start in readonly mode. ttyd v1.7.x defaults to read-only; the -W flag is required for interactive use. The work wiki had this; it was missed during adaptation.

Entry 4 — Terminal working directory

Third friction point. The shell inside the terminal started in _app/ instead of the wiki root (/home/sprite). This matters because claude --resume needs to run from the project root to find the correct session. Root cause: the wiki up command was cd’ing to $APP_DIR for the node server launch, and ttyd inherited that directory. Fix: cd to $WIKI_ROOT before launching ttyd, and launch the node server via absolute path instead.

Entry 5 — Tree walker: directories with only index.html

The nav tree was missing all four workstream directories. Root cause: the tree walker filtered out directories with zero children, but a workstream directory whose only file is index.html consumes that file as the directory’s title and skips it from the children list, leaving an empty array. Fix: track whether a directory has an index.html via a hasIndex flag, and include directories that have one even if they have no other children.

Entry 6 — Sprite services instead of raw PID management

The first version of the wiki CLI managed two background processes (node server, ttyd) by hand — spawning, writing PIDs to _app/run/, probing ports until listening. That works locally but the wiki was unreachable from outside the Sprite VM: the Sprite proxy at https://<sprite>.sprites.app routes external traffic to a service registered with --http-port, not to whatever happens to be listening on a port. No service, no route.

Fix: register ttyd and the node server as separate Sprite services with sprite-env services create. The wiki service declares --http-port 3000 (so the proxy routes there) and --needs ttyd (so dependency ordering is enforced). Two payoffs: external reachability, and auto-restart on crash — the supervisor brings the process back without the CLI having to track PIDs.

One gotcha: --args splits on commas. The ttyd font family arg fontFamily=Menlo,Monaco,Consolas,monospace got shredded into separate args; the start command became “Monaco\ Consolas\ monospace zsh” and ttyd tried to execvp a file named “Monaco\”. Dropped the font family from the service args (it’s cosmetic) and added fontSize=16 separately.

The wiki CLI (up/down/status) now wraps sprite-env services start/stop/list instead of doing the bookkeeping itself. Less surface area, fewer corners that go wrong.

Entry 7 — Browser-to-VM image bridge

ttyd is a text terminal — it cannot receive image pastes. But the wiki’s primary mode is “Claude Code running in the terminal pane,” and Claude Code reads images via file paths. So the operator’s real need is to get a screenshot from the MacBook clipboard into the VM filesystem, with the path on the clipboard for pasting into the terminal.

Solution: a drop/paste zone in the topbar. Cmd+V or drag-drop sends the image to POST /api/upload, the server writes it to _uploads/screenshot-<ts>.png, returns the absolute path, and the frontend copies that path to the OS clipboard via navigator.clipboard.writeText. The operator pastes the path into Claude Code and asks it to look.

Placement matters: the drop zone lives in the topbar header, not the content pane. Wiki pages are intended to be little applications — they may have their own paste handlers, drop targets, or interactive surfaces. Putting the bridge in topbar chrome keeps the content pane free for pages to be themselves.

Entry 8 — Server self-restart via process.exit

The terminal proxies through the node server. Restarting node from the terminal’s own Claude Code session kills the proxy connection, stranding the operator with no terminal and no easy way to recover. A topbar ⟳ button is needed so a restart can be triggered from the wiki page (which survives the restart) instead of the terminal (which doesn’t).

First attempt: POST /api/restart handler spawns sprite-env services restart wiki via execFile. Failed with open /sys/fs/cgroup/containers/sprite/svc.wiki/cgroup.procs: No such file or directory. A service cannot restart itself through sprite-env — the supervisor’s cgroup machinery doesn’t expect the target service to be the caller. The service hung in a half-broken state (PID reported as running, port not listening) until manually restarted.

Correct pattern: the handler returns 200, then calls process.exit(0). Sprite’s supervisor detects the exit and auto-restarts the service within ~2 seconds (see Entry 6). The frontend polls /api/tree until it gets 200, then reloads the terminal iframe. The operator stays in one browser tab; they need claude --resume in the new terminal session to pick up where they left off.

General lesson: when a process needs to restart itself, exit cleanly and let the supervisor handle it. Don’t reach back through the supervisor’s control plane from inside the supervised process.

Entry 9 — /checkpoint skill: commit messages as nucleus input

Two persistence layers run in parallel: git (durable, version- controlled) and Sprite checkpoints (full VM-state rollback). A milestone usually wants both, so doing them one-at-a-time is friction. The /checkpoint skill bundles them: survey changes, draft a commit message, commit, then create a Sprite checkpoint with the same subject line.

The interesting discipline isn’t the bundling — it’s the commit-message bar. Entry 0 establishes /evening reads git history to propose nucleus updates. That makes the commit message the nucleus’s primary input: every commit subject + body should capture the decision and reasoning, not just what files were touched. A “Fix bug” commit costs the nucleus a future entry it could have had.

The skill encodes the bar with positive and negative examples inline (good: “Register wiki as Sprite services instead of raw PID management” with the proxy-routing explanation; bad: “Update files”). When a change bundles multiple distinct decisions, the skill suggests splitting into separate commits so each decision gets its own message.

Entry 10 — The nuclei system: move the nucleus into a shared, public repo

The nucleus was sitting at personal-nucleus.html inside the wiki repo. That made it private to this sprite, hard to share, and impossible to steal from — the captain’s log model only pays off when the log is discoverable to other systems. The shift: move the nucleus to its own public repo (github.com/rphilander/nuclei), cloned locally to /home/sprite/nuclei/, with the wiki repo recording the file’s deletion and the nuclei repo owning history going forward.

The repo holds multiple nuclei (one per system) plus a CLAUDE.md spec, a CC BY 4.0 LICENSE, and a skills/ directory of shared slash-command skills. The conceptual model is three pieces: the local system (a project with its own git repo, like this wiki), the local nucleus (its decision log, one HTML file in the nuclei repo), and the remote nuclei (every other system’s log, available to learn from or steal from).

The bridge between a sprite and the nuclei repo is a .nuclei.json config at the sprite’s root, declaring the local system’s identity, nucleus filename, and both repo paths. Shared skills read this config to auto-discover. Skills (/nucleus-update, /nucleus-scan-for-steals) live in the nuclei repo and are symlinked into each sprite’s ~/.claude/skills/ — symlinks themselves are gitignored as per-sprite installation artifacts.

The local /evening skill became a thin wrapper that delegates to /nucleus-update. The pattern is: system-specific orchestrators (/morning, /evening) stay local on each sprite and own the system-specific intel sources and judgment; shared skills handle the nucleus-protocol mechanics. Same intent as the proposal pattern from work Entry 19 — separate the judgment from the placement.

Cross-sprite idea-stealing flows through this surface: /nucleus-scan-for-steals pulls the nuclei repo, compares each remote nucleus against a per-sprite cursor at ~/.config/nuclei-cursor.json, surfaces new entries with steal/skip/defer dispositions, and hands confirmed steals to /nucleus-update for placement with an attribution cross-reference. The nucleus is the feed — no separate RSS/webhook layer needed.

Entry 11 — System-coupled rules belong in the repo, not in private memory

Claude Code has a built-in memory mechanism — structured per-rule files at .claude/projects/<encoded-dir>/memory/, each with frontmatter (name, description, type) and a Why / How-to-apply discipline. The harness auto-loads MEMORY.md as an index; individual memory files are consulted by Claude on demand when they seem relevant. That location was wholesale gitignored, which made every memory file private to one Claude instance on one sprite.

The bug in framing: not all memory is operator-private. Some rules are system-coupled — they capture how Claude must behave to operate the system correctly, and they should propagate with the system to any sprite or any future Claude. Treating those as private was costing portability.

The reframe: memory holds two kinds of content. (1) Operator-private preferences — actually personal, no system attached. (2) System-coupled behavioral rules — integral to how this system works. The format is the same; the appropriate storage is not. Mechanism: keep the harness’s memory directory exactly where it is, but un-gitignore it. A narrow gitignore exception (!.claude/projects/-home-sprite/memory/) tracks the memory files while leaving session transcripts and subagent state ignored. Genuinely private memories later can be ignored individually.

One subtlety about how memory loads: the index is auto-loaded each session, but individual memory files are only consulted when Claude judges them relevant. For hard rules (don’t bash-restart the wiki), that’s not enough — you want the rule to fire before the action, not be retrieved after-the-fact. Workaround: the local CLAUDE.md (auto-loaded in full, every session) gets a brief “Operating discipline” section that prompts Claude to consult the memory directory before any disruptive action. The discipline section is the trigger; the memory file is the content.

Three layers, no redundancy: CLAUDE.md for orientation and triggers, memory/ for structured granular rules (now in the repo), the nucleus for the durable, public, captain’s-log record of decisions (cross-references back to memory where applicable). Memory and nucleus describe the same content at different audiences: memory is for Claude operating now, the nucleus is for Claude (or human) understanding later.

Entry 12 — The button is the only restart path — bash kills more than ttyd

Entry 8 got the technical mechanism right (server self-restart via process.exit(0) handled by Sprite’s supervisor) but undersold the operational consequence. The lesson learned through experience today: any bash-initiated restart of the wiki service — including the sprite-env services restart wiki that produced Entry 8’s cgroup error — cascades farther than expected.

The cascade: bash restart kills the node server → that breaks the ttyd reverse-proxy WebSocket → the terminal pane’s shell session terminates → whatever Claude Code session was running in that terminal is killed mid-turn → conversation-history-to-disk has a race where the final turns of the dying session can be silently dropped from the persisted transcript. The operator gets stranded and loses context.

The protocol that follows: never bash-restart the wiki service. Always ask the operator to click the ⟳ (restart) button in the wiki topbar, which calls process.exit(0) from the server itself and lets the frontend deterministically poll and reload the terminal iframe. The terminal session still ends (the iframe reloads), but the operator stays in one browser tab, can run claude --resume, and the history-write race doesn’t affect the new session.

Only acceptable exception: emergency recovery when the server is in a half-broken state and the button isn’t reachable (e.g., 502 Bad Gateway). After such a recovery, return to the button-only protocol.

The rule lives operationally as memory/feedback_wiki_restart.md (per Entry 11) and is referenced by the Operating discipline section of CLAUDE.md so it fires before the action, not after. The lesson here is broader than restarts: any action that traverses the reverse-proxy can cascade through the terminal session and the writeable conversation buffer. “Just restart it” is rarely “just.”

Entry 13 — Live-reload via chokidar: filesystem-as-database needs filesystem-as-event-source

The wiki treats the filesystem as the database (per work Entry 0) and edits to content files are the normal way the system evolves. Without live reload, every edit required a manual page refresh to see the result — tolerable for a one-off but corrosive over the course of a session because the feedback loop is interrupted at exactly the moment you want to verify a change. The work LLM wiki had filesystem watching from early on and it didn’t make the port over. Today it did.

Implementation choice: chokidar over Node’s built-in fs.watch. Reason: fs.watch’s recursive option is unreliable on Linux (silently drops events, inconsistent for nested directory creation). chokidar normalizes across platforms and handles the corner cases that would otherwise have to be re-invented. The trade-off is one dependency for a behavior that’s table stakes.

Two parameters that matter operationally. (1) Ignored patterns: dotfiles, node_modules, _app/run/, _uploads/. Without these, the watcher fires constantly on session-state files Claude writes (in .claude/) and on PID/log files the wiki service rewrites. (2) Debounce + awaitWriteFinish. Editor saves typically produce a burst of FS events (temp file, atomic rename, etc.); coalescing into a single broadcast at the end prevents reload thrash.

The frontend already had a reload message handler in the WebSocket from earlier; the new code just wires chokidar.on('all') to broadcast({type:'reload'}) through the existing wsClients set. That meant the server-side change was tight and the client-side change was nothing — the capability was latent, waiting for a trigger.

Consequence: the manual ↓ refresh button in the topbar became redundant and was removed. The ⟳ server-restart button stays — different purpose (forces the node server to exit and Sprite to bring it back up, per Entry 8). Two buttons that looked similar but were doing different jobs; collapsing them clarifies the affordance.
