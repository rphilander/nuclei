Preface

This document captures the decisions and reasoning behind truespeech-console — a filesystem-served three-pane viewer (left nav, content, terminal) plus an image bridge, running on the truespeech-lexicon sprite to support active development of the TrueSpeech language and lexicon. The system mirrors the personal LLM wiki's architecture (Node+Express on port 3000, vanilla JS frontend, ttyd reverse-proxied at /term/*, registered as sprite services) but is scoped narrowly to the TrueSpeech development work: viewing demo pages, browsing the source repos, maintaining a roadmap and bug registry, and bridging screenshots from the operator's MacBook clipboard into the VM filesystem.

The system was adapted from the personal LLM wiki's nucleus (personal.html) with deliberate omissions — no workstream hierarchy, no action items, no proposal pattern, no /morning or /evening orchestrators, no showcase index, no two-zones split. Those were features specific to the LLM-wiki framing; the console's use case is narrower. Most of what carried over is the infrastructure (Express + ttyd + sprite services + same-origin iframe + chokidar live-reload + image bridge) rather than the wiki-specific surface.

The audience for this nucleus is a future Claude (or other model) rebuilding, extending, or stealing from the system. Entries accumulate as decisions get made; the expected cadence is "most days nothing nucleus-worthy happens." /nucleus-update reads recent git activity and proposes entries when something crosses the bar.

Entry 0 — Static-serve from /home/sprite, not /api/page, for HTML demos

The personal LLM wiki's /api/page endpoint (personal Entry 1) served every kind of content — HTML, Markdown, CSV, images — wrapped in a single response. That worked because wiki content was small, self-contained pages. It would break here: the primary content the console needs to show is the truespeech.github.io demo (regions.html, truespeech.html), and those pages load their own JS modules and CSS via relative paths (truespeech/dist/main.js, images/truespeech-architecture.png, etc.). If the iframe URL is /api/page?path=/truespeech.github.io/regions.html, the relative imports resolve against /api/page?path=/... and produce 404s on every asset.

The fix is a split. express.static('/home/sprite', { dotfiles: 'ignore' }) serves any file under the home tree at its actual URL path. HTML files load via that mount — the iframe points at /truespeech.github.io/regions.html directly and the relative imports work. /api/page handles only the things that need transformation: Markdown via markdown-it, source files highlighted via highlight.js, plain-text fallback for everything else. The frontend routes per file extension — HTML goes to the static URL, everything else through /api/page.

Two consequences worth naming. (1) express.static's dotfiles: 'ignore' option is load-bearing — without it, GET /.github_token would happily return the PAT. The nav filter already hides dotfiles client-side, but the HTTP layer needs its own independent guard. (2) /api/page still validates the request path with safePath() (resolve against HOME, reject anything that escapes) per work Entry 1, even though static-serve is the more common surface. Both paths get the discipline.

The wider lesson: when the content the system serves is itself code that loads more code, the URL the browser sees has to be the URL the code expects. Wrapping in an API endpoint is fine for inert content; not fine for live applications.

Entry 1 — ttyd and tsconsole as separate sprite services, not one

Per personal Entry 6, sprite services give external reachability (via --http-port) and auto-restart on crash. Here we register two services rather than one: ttyd (the terminal backend) and tsconsole (the Node server), with --needs ttyd on the latter so the supervisor starts them in dependency order. Only tsconsole declares --http-port 3000 — ttyd has no externally visible port; it's reachable only through tsconsole's same-origin reverse-proxy at /term/*.

Two payoffs. (1) Each service has its own crash-recovery loop — when the Node server self-restarts via process.exit(0) (lifted from personal Entry 8), ttyd stays up; the embedded terminal reconnects automatically once the proxy comes back. (2) The operator's browser sees exactly one URL surface (truespeech-lexicon-bk5b2.sprites.app) — the ttyd port is implementation detail.

One --args gotcha worth recording so future-Claude doesn't re-step on it (already documented in personal Entry 6, but it bit hard enough to mention again): sprite-env splits the args list on commas, so any individual arg value containing a comma gets shredded into separate args. The ttyd font-family arg fontFamily=Menlo,Monaco,Consolas,monospace would have become four args; the workaround was to skip the font family entirely (it's cosmetic) and pass only fontSize=14 along with rendererType=canvas (work Entry 16's pin).
